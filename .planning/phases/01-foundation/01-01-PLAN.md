# Phase 1 Implementation Plan: Foundation & Core Vision

## Goal
Establish the technical foundation, focusing on background operation capability and the core overlay UI system.

## Requirements Traceability
- **BACK-01**: System shall run continuously in background while foreground navigation apps active.
- **BACK-02**: System shall provide overlay UI visible above other applications.
- **BACK-03**: System shall maintain <10% CPU usage during background operation.
- **BACK-04**: System shall maintain <50MB memory footprint during background operation.

---

## Wave 1: Core Background Service Architecture
**Focus**: Ensuring the application remains alive and operational when the user switches to a navigation app (Google Maps, Waze, etc.).

### Task 1.1: Implement Android Foreground Service
- **Description**: Create a persistent Foreground Service to prevent the OS from killing the app process during background navigation.
- **Technical Steps**:
    1. Define a `ForegroundService` class extending `Service`.
    2. Implement `onStartCommand` to handle service lifecycle.
    3. Create a high-priority persistent notification (required for Foreground Services) to notify the user that Guardian AI is monitoring.
    4. Configure `AndroidManifest.xml` with `FOREGROUND_SERVICE` and `FOREGROUND_SERVICE_SPECIAL_USE` (or appropriate type) permissions.
- **Verification Criteria**: Launch the app, start the service, switch to Google Maps for 10 minutes. Verify the app process is still running via ADB/Logcat.
- **Success Measure**: 0% unplanned process terminations during 30-minute navigation sessions.
- **Satisfies**: BACK-01

### Task 1.2: Background Permission Orchestration
- **Description**: Implement a permission request flow to ensure all necessary system-level permissions are granted.
- **Technical Steps**:
    1. Implement a "Permissions Dashboard" UI to explain why background and overlay permissions are needed.
    2. Request `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` to prevent aggressive Doze mode killing.
    3. Request `SYSTEM_ALERT_WINDOW` (Draw over other apps) permission.
    4. Implement a check-and-redirect logic to System Settings if permissions are denied.
- **Verification Criteria**: Verify that the app correctly detects missing permissions and guides the user to the specific settings page.
- **Success Measure**: 100% of required permissions granted before service start.
- **Satisfies**: BACK-01, BACK-02

## Wave 2: Overlay UI & Presentation Layer
**Focus**: Creating a non-intrusive yet visible UI layer that sits atop the navigation app.

### Task 2.1: Overlay Window Management
- **Description**: Implement the `WindowManager` logic to render a custom view on top of all other apps.
- **Technical Steps**:
    1. Use `WindowManager.LayoutParams` with `TYPE_APPLICATION_OVERLAY`.
    2. Set flags for `FLAG_NOT_FOCUSABLE` to ensure the navigation app beneath still receives touch events (unless the overlay is specifically interacted with).
    3. Implement logic to dynamically show/hide the overlay based on system state.
    4. Handle screen rotation and resize events to maintain overlay positioning.
- **Verification Criteria**: Verify that a test "Guardian Indicator" is visible while Google Maps is active. Verify that the user can still interact with Google Maps buttons.
- **Success Measure**: Overlay renders in <200ms upon service trigger.
- **Satisfies**: BACK-02

### Task 2.2: High-Contrast Overlay Components
- **Description**: Design and implement the visual elements of the overlay using high-visibility design principles.
- **Technical Steps**:
    1. Implement a "Safe State" indicator (e.g., small green dot/icon).
    2. Implement the "Warning State" container (e.g., pulsing red border for the screen edges).
    3. Ensure color contrast ratios meet accessibility standards (WCAG AAA) for sunlight visibility.
    4. Use hardware-accelerated drawing for animations to minimize CPU impact.
- **Verification Criteria**: Visual inspection of the overlay under simulated bright light conditions.
- **Success Measure**: Visuals are clearly distinguishable from navigation app UI.
- **Satisfies**: BACK-02

---

## Wave 3: Resource Optimization & Benchmarking
**Focus**: Tuning the foundation to meet strict CPU and Memory constraints.

### Task 3.1: Resource Profiling & Baseline Measurement
- **Description**: Establish a baseline for CPU and Memory usage during idle background operation.
- **Technical Steps**:
    1. Use Android Studio Profiler / ADB shell `top` and `dumpsys meminfo` to capture metrics.
    2. Measure CPU usage across different device tiers (Low, Mid, High end).
    3. Measure Heap and Native memory allocation during the "idle background" state.
- **Verification Criteria**: Logged metrics for CPU and RAM over a 1-hour period.
- **Success Measure**: Baseline documented for comparison.
- **Satisfies**: BACK-03, BACK-04

### Task 3.2: Performance Tuning & Optimization
- **Description**: Optimize the service and overlay to stay within the <10% CPU and <50MB RAM limits.
- **Technical Steps**:
    1. Optimize `WindowManager` update frequency (avoid unnecessary redraws).
    2. Implement lazy loading for overlay resources.
    3. Audit for memory leaks using LeakCanary.
    4. Optimize notification updates to avoid excessive IPC calls.
- **Verification Criteria**: Re-run profiling from Task 3.1.
- **Success Measure**: CPU usage < 10% (Target < 5%) and Memory < 50MB during idle background state.
- **Satisfies**: BACK-03, BACK-04

---

## Implementation Sequence
1. **Task 1.1** $\rightarrow$ **Task 1.2** (Get the service running and permitted)
2. **Task 2.1** $\rightarrow$ **Task 2.2** (Get the UI visible and polished)
3. **Task 3.1** $\rightarrow$ **Task 3.2** (Ensure the system is lightweight)
